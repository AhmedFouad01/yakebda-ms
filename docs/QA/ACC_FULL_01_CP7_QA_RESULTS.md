# ACC-FULL-01 — نتائج مصفوفة QA اليدوية (CP7)

**التاريخ:** 2026-07-21
**المُختبِر:** Claude Code (تشغيل يدوي عبر المتصفح)
**البيئة:** قاعدة disposable معزولة `ykms_cp7` (createdb → migrate:latest (حتى 028) → seedFoundation → fixtures عبر المسارات الفعلية). **لم تُلمس `ykms` الأصلية ولا `ykms_test`.** الـAPI مربوط بـ`ykms_cp7` عبر `DATABASE_URL` (تأكيد العزل: مستخدمو QA يوجدون في `ykms_cp7` فقط ونجح دخولهم).
**القياسات المُختبَرة:** 1440×900 و390×844 (mobile)، في Dark + Light.
**بيانات الاختبار:** 7 حالات أحداث (posted/failed/dead/pending/pending_policy/non_posting/deferred_rounding)، 4 فروق تقريب مفتوحة (0.0160)، قيود يدوية + عكس، شجرة ADR-004 كاملة بحساب التقريب 4090.

> نتيجة نهائية: **كل البنود Pass** بعد إصلاح Fail‑ين مكتشفين (انظر قسم الإصلاحات). ملاحظة صدق: النجاح = جاهزية ضمن النطاق، وليس اعتمادًا statutory/production (يبقى محكومًا بـADR-009/FR-289).

## الإصلاحات المطبَّقة في CP7 (نتيجة QA)

1. **شاشة «الإعدادات» (Fail قسم 9):** لم تُبنَ في CP5/CP6 رغم وجود العقد GES/PUT `/accounting/settings` (CP3) وإشارة ADR-004 لها. أُضيفت شاشة تحرير القيم القابلة للتعديل (النوع ب) + عرض ثوابت النوع أ للقراءة، محكومة بـ`accounting.manage`.
2. **تكرار صفوف الإعدادات على مستوى الحساب (Fail):** PostgreSQL يعامل `branch_id = NULL` كقيمة متمايزة في unique index، فلم يعمل `onConflict().merge()` وأنشأ صفًا جديدًا في كل حفظ. أُصلح بـupsert آمن للـNULL (delete-scope ثم insert داخل transaction). اختبار regression: 3 حفظات → صف واحد.
3. **حزمة المراجعة — «التسويات وعكوسها»:** كانت تجلب `residual.settlement` فقط؛ أُضيف جلب عكوس التسويات (journal.reversal ذات المصدر = قيد تسوية) ودمجها لتصبح القائمة أمينة للعنوان.
