# وثيقة متطلبات النظام SRS — Restaurant MS v1.3 AR/RTL

**التاريخ:** 2026-07-05  
**الإصدار:** v1.3  
**اللغة الأساسية:** العربية  
**اتجاه الواجهة:** RTL من اليمين إلى الشمال  
**نوع النظام:** Restaurant Management System / POS / RMS  
**حالة المشروع:** مشروع مستقل بالكامل عن AKYRO  

---

## 1. الملخص التنفيذي

نظام **Restaurant MS** هو نظام إدارة مطاعم متكامل، عربي أولًا، ومتوافق مع أجهزة Windows، ويهدف إلى تشغيل عمليات المطعم اليومية من نقطة واحدة.

النظام ليس مجرد Dashboard، بل منصة تشغيل تشمل:

- كاشير POS.
- إدارة الطلبات.
- إدارة الطاولات.
- شاشة المطبخ KDS.
- الطباعة للمطبخ والعميل.
- إدارة المنيو والأسعار والإضافات.
- إدارة الشيفتات والكاش.
- إدارة العملاء.
- إدارة المخزون والوصفات.
- التقارير.
- API للموقع، QR، والتطبيقات المستقبلية.
- طبقة ربط محلية مع الهاردوير على أجهزة Windows.

الهدف الأول هو بناء نسخة MVP عملية تصلح لمطعم واحد أو عدة فروع لاحقًا، مع تصميم معماري يسمح بالتوسع بدون إعادة بناء النظام من الصفر.

## 2. القاعدة الأساسية للغة والاتجاه

### 2.1 اللغة

- اللغة الأساسية في الواجهة: **العربية**.
- كل النصوص الظاهرة للمستخدم يجب أن تكون بالعربية في النسخة الأولى.
- الإنجليزية يمكن دعمها لاحقًا كلغة اختيارية.
- أسماء الجداول، الـ APIs، المتغيرات، والـ commits يمكن أن تكون بالإنجليزية لتسهيل التطوير.

### 2.2 اتجاه الواجهة

- الاتجاه الافتراضي لكل الشاشات: **RTL**.
- يجب ضبط `dir="rtl"` على مستوى التطبيق العربي.
- يجب أن تكون القوائم الجانبية، الجداول، النماذج، أزرار الإجراءات، الفلاتر، ورسائل التنبيه مناسبة لاتجاه RTL.
- عند دعم الإنجليزية لاحقًا، يتحول الاتجاه إلى LTR فقط عند اختيار اللغة الإنجليزية.

### 2.3 قواعد UI عربية

- الأرقام يمكن عرضها إنجليزيًا أو عربيًا حسب إعداد النظام، لكن يفضل دعم الخيارين.
- التاريخ والوقت يجب أن يدعما تنسيق مصر/المنطقة العربية.
- أسماء المنتجات، الفئات، الإضافات، والطاولات تظهر بالعربي افتراضيًا.
- رسائل الأخطاء والتنبيهات يجب أن تكون واضحة وبسيطة باللغة العربية.

## 4. نطاق المشروع

### 4.1 نطاق النسخة الأولى MVP

النسخة الأولى يجب أن تشمل:

- لوحة إدارة عربية RTL.
- شاشة POS للكاشير على Windows.
- إدارة المستخدمين والصلاحيات.
- إدارة الفروع والأجهزة.
- إدارة المنيو الأساسية.
- إدارة الطلبات الأساسية.
- إدارة الطاولات.
- شاشة مطبخ KDS.
- طباعة إيصال العميل وتذكرة المطبخ.
- إدارة المدفوعات الأساسية.
- إدارة الشيفت والكاش.
- العملاء الأساسيين.
- مخزون ووصفات بشكل أولي.
- تقارير يومية أساسية.
- API داخلي و Public API v1 للموقع والـ QR لاحقًا.
- Local Device Bridge مبدئي للتعامل مع الهاردوير على Windows.

### 4.2 خارج نطاق النسخة الأولى

هذه العناصر تؤجل بعد MVP:

- تطبيق موبايل Native.
- برامج ولاء متقدمة.
- تكامل مباشر مع بوابات دفع متعددة.
- تكامل ضريبي/فاتورة إلكترونية نهائي قبل تحديد الدولة والمتطلبات القانونية.
- Offline-first Sync كامل.
- BI متقدم وتوقعات مبيعات.
- Marketplace كامل للتكاملات.

## 6. الافتراضات التقنية

- التطبيق الأساسي Web App / PWA يعمل على Chrome أو Edge في وضع Kiosk على Windows.
- يمكن لاحقًا تغليف شاشة POS بتقنية Electron أو Tauri لو احتجنا تحكم أقوى في الهاردوير.
- الاتصال بالطابعات ودرج الكاش يتم عبر Local Device Bridge يعمل على جهاز Windows داخل الفرع.
- النظام يدعم Cloud Backend، ويمكن لاحقًا إضافة Offline-first أو Local Server حسب الحاجة.
- الطابعات قد تكون USB أو LAN أو Bluetooth حسب بيئة المطعم.
- يجب دعم ESC/POS أو Windows Printer Drivers حسب نوع الطابعة.
- الالتزام الضريبي والفاتورة الإلكترونية يحتاج قرار منفصل حسب الدولة قبل الإنتاج النهائي.
