# تعليمات العمل في المشروع — YAKEBDA MS v1.0 AR/RTL

هذه الوثيقة إلزامية لأي جلسة عمل أو مساهمة في المشروع.

---

## 1. قواعد الهوية والتسمية

- اسم المشروع في أي كود أو وثيقة أو واجهة: **YAKEBDA MS**.
- اسم المستودع/المجلد: `yakebda-ms` — أسماء الحزم: `@ykms/api`، `@ykms/admin`، `@ykms/bridge-contract`.
- تسمية المراحل: `YKMS-XX` فقط.
- **ممنوع** استخدام: Restaurant MS، restaurant-ms، RMS-XX كتسمية نشطة — هي تاريخية فقط داخل وثائق SRS/Diagrams الأصلية.
- **ممنوع تمامًا** خلط المشروع مع أي ملفات أو branches أو prompts أو roadmap خاصة بـ AKYRO.
- بادئة رموز API: `ykms_` — مفتاح تخزين الجلسة في المتصفح: `ykms_token`.

## 2. قواعد اللغة والاتجاه

- كل النصوص الظاهرة للمستخدم **بالعربية** عبر كتالوجات الترجمة (`apps/api/src/i18n/ar.ts` و `apps/admin/src/lib/t.ts`) — لا نصوص hardcoded.
- الاتجاه الافتراضي `dir="rtl"` على مستوى `<html>`.
- أسماء الجداول والـ APIs والمتغيرات والـ commits بالإنجليزية لتسهيل التطوير.
- التاريخ والوقت بتنسيق `ar-EG`.
- أي شاشة جديدة تُبنى وتُختبر بالعربي أولًا؛ الإنجليزية لاحقًا كخيار وليست أساسًا.

## 3. قواعد النطاق

- يُنفذ فقط ما تحدده المرحلة الجارية في `docs/YAKEBDA_MS_Milestone_Log.md`.
- المؤجل حاليًا وممنوع تنفيذه قبل مرحلته: المنيو، الطلبات، المدفوعات، المخزون، التقارير، تطبيق الموبايل.

## 4. قواعد الأمان (إلزامية في أي كود جديد)

1. أي مسار مستخدم يمر عبر `requireUser` + `requirePermission` عند الحاجة.
2. أي مسار token يمر عبر `requireApiToken` مع الـ scope الصحيح.
3. **عزل الحسابات:** أي استعلام قراءة أو تعديل يُقيد بـ `account_id` الخاص بالمستخدم أو الـ token — عبر العمود مباشرة أو عبر join على `branches.account_id`. لا يُقبل أبدًا استعلام بـ id فقط.
4. أي ربط بين كيانين (مثل endpoint ↔ device) يتحقق أن الطرفين في نفس الحساب ونفس الفرع عند اللزوم.
5. الرموز تُخزن hash فقط (sha256) وتُعرض مرة واحدة عند الإنشاء.
6. كل إجراء حساس يُسجل في Audit Log مع المستخدم/العميل والفرع والجهاز.
7. أي تعديل أمني يُرفق باختبار يثبت منع الوصول العابر للحسابات.

## 5. قواعد الاختبار والتسليم

- الاختبارات في `apps/api/tests/` — تعمل على قاعدة `ykms_test` عبر `NODE_ENV=test npx vitest run`.
- لا تُعتبر مرحلة مكتملة إلا باختبارات خضراء تغطي معايير قبولها.
- كل مرحلة تُقفل بتحديث `docs/YAKEBDA_MS_Milestone_Log.md`.
- القرارات المعمارية تُوثق كـ ADR في `docs/adr/`.

## 6. أوامر التشغيل المرجعية

```bash
./scripts/setup-db.sh        # قواعد ykms + ykms_test
npm install
cp apps/api/.env.example apps/api/.env
npm run api:migrate && npm run api:seed
npm run api:dev              # المنفذ 3001
npm run admin:dev            # المنفذ 5173
npm run api:test             # الاختبارات
```

بيانات تجريبية: `owner@ykms.local / Owner@12345` — كاشير PIN `1234`.
