# ADR-001 — عزل الحسابات في طبقة الجسر + توحيد هوية YKMS

**التاريخ:** 2026-07-05 — **الحالة:** مقبول

## السياق

1. مسارات Local Device Bridge كانت تستعلم عن الأجهزة ومهام الطباعة ونقاط الهاردوير بالـ `id` فقط، مما كان يسمح نظريًا لأي token يحمل نطاق `bridge` — من أي حساب — بقراءة أو تعديل بيانات حساب آخر.
2. كذلك كان إنشاء نقطة هاردوير يقبل `device_id` دون التحقق من ملكيته.
3. جدول `print_jobs` لا يحمل `account_id` مباشرًا.
4. بالتوازي، تم اعتماد الهوية الرسمية YAKEBDA MS بدل الاسم التاريخي Restaurant MS.

## القرار

1. **العزل عبر السلسلة الموجودة بدل إضافة عمود:** يُشتق حساب أي مهمة طباعة من `print_jobs.branch_id → branches.account_id` عبر join، ويُشتق حساب الجهاز من عمود `devices.account_id` الموجود. لا حاجة لـ migration جديد في هذه المرحلة، والفهارس الحالية كافية لحجم MVP. (يمكن إعادة النظر بإضافة `account_id` غير قابل للتغيير denormalized على `print_jobs` لاحقًا لو ظهرت حاجة أداء.)
2. **قاعدة إلزامية:** أي مسار token أو مستخدم لا يستعلم أبدًا بـ `id` وحده؛ دائمًا بـ `id + account_id` أو join مكافئ. أي ربط بين كيانين يتحقق من تطابق الحساب — وللهاردوير تطابق الفرع أيضًا.
3. **سلوك الفشل:** الوصول العابر للحسابات يرجع `404` (لا `403`) حتى لا يكشف وجود الكيان، وتحديثات heartbeat على نقاط خارج الحساب تُتجاهل بصمت مع نجاح باقي الطلب.
4. **الهوية:** التسمية النشطة الوحيدة YAKEBDA MS / `yakebda-ms` / `@ykms/*` / `YKMS-XX`، وبادئة الرموز `ykms_`. تغيير البادئة آمن الآن لأنه لا توجد رموز إنتاجية صادرة؛ بعد الإنتاج يتطلب أي تغيير مشابه فترة قبول مزدوج.

## النتائج

- 7 اختبارات عزل جديدة خضراء، ولا تغيير في عقد الجسر الخارجي (نفس المسارات والحمولات).
- كلفة join إضافية طفيفة على مسارات الجسر — مقبولة.
- الوثائق التاريخية (SRS/Diagrams باسم Restaurant MS) تبقى دون تعديل كمرجع.
